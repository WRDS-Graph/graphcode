## (2) The specific seam to split along

The class has four cleanly separable concerns. The honest constraint: **all the mutating methods share `this.cy` and the 24 state fields**, so I will *not* shatter the class into many classes (that forces threading state through constructors and breaks the test's `getCyInstance` assumptions). Instead, split along the seam between **stateless / config code** (pull fully out to modules) and **stateful orchestration** (keep in the class, but thinned).

**Seam A — Pure, zero-`this` code → extract to standalone modules (no behavior risk):**

- `src/renderer/graphStylesheet.ts` ← `buildStylesheet()` + the `SANS/SERIF/MONO` font consts (~480 lines, the single biggest chunk, touches no state).
- `src/renderer/graphConstants.ts` ← `GROUP_META`, `ALL_GROUPS`, `MAX_FAN_SIZE`, `DEFAULT_RENDER_OPTIONS`, `NodeType`, `RenderOptions`.
- `src/renderer/graphModel.ts` ← `deriveNodeType`, `buildChildrenMap`, `buildEdgeTypeMap`, `formatEdgeType`, `computeImportance` (pure functions over `PaperEdge[]`/`PaperNode`).

That alone moves **~700+ lines out** with essentially no risk, because none of it reads `this`.

**Seam B — Layout math → extract to a pure module:**

- `src/renderer/graphLayout.ts` ← the fan/positioning geometry currently inlined in `renderClusterPage`, `toggleNodeExpansion`, `renderCompClusterPage` (the `baseAngle`/`fanSpread`/`fanRadius`/`stemLength` arithmetic) and the overlap solver body of `resolveOverlaps()` (lines ~1424–1452). Extract as pure functions: `computeFanPositions(origin, baseAngle, count) → {x,y}[]` and `resolveOverlaps(boxes) → positions[]`. The class methods keep the Cytoscape `.position()`/`.show()` calls but **delegate the math**. This is the highest-value extraction for testability.

**Seam C — Comparison-side logic stays grouped but split by file via a `// @ts-expect`-free mixin or a second source file using `declare module` augmentation is *not* worth it.** Instead, the remaining `CytoscapeGraphRenderer` keeps all `this.cy`-mutating methods but is now ~1800 lines focused purely on stateful orchestration: render lifecycle, cluster/node expand-collapse, comparison overlay, drag, interactions.

If further reduction is wanted later, the comparison methods (`toggleCompCluster`, `renderCompClusterPage`, `pageCompCluster`, `toggleCompNodeExpansion`, `renderSharedComparisonCluster`, the `comp*` state fields) form a natural future second class — but defer that; it's the riskiest cut because comparison and canonical state interleave in `renderOverlay`.

**Why this seam and not "one class per concern":** The class is a god-object by *size*, not by *coupling surface* — its fan-in is 2 files. The cheap, safe win is pulling out the **stateless 700–1000 lines** (style + constants + pure helpers + layout math), which shrinks the file by ~⅓ with near-zero regression risk and gives the layout math its own unit tests. Re-architecting the stateful core into multiple collaborating classes is high-risk for low marginal benefit and should be a separate, later step.

## (4) Top risk

**The overlap/fan layout math in Step 5 is positional and only weakly pinned by tests.** The geometry (`resolveOverlaps`, the `baseAngle`/`fanRadius`/`stemLength` arithmetic) produces exact `x,y` coordinates that feed Cytoscape `.position()`. The test suite asserts on **visibility and `isHidden` data flags** far more than on exact coordinates, so a subtle off-by-one in the extracted math (e.g. dropping the `this.currentOverlay !== null && meta.dx < 0` angle-flip at line 1161, or the overlap solver's `if (xs[j] >= xs[i])` tie-break at line 1434) can pass all unit tests while visibly breaking the layout — nodes overlapping, fans pointing into the comparison graph, or the documented **Issue #9 zoom/pan regression** (CLAUDE.md §6.4 invariant: preserve `cy.zoom()`/`cy.pan()` across mutations; `pageCluster` deliberately avoids `cy.fit()` — line 1227–1232).

**Mitigation:** Treat Step 5 as behavior-preserving with character-level fidelity — move the arithmetic verbatim into pure functions (don't "clean it up" simultaneously), add unit tests asserting the extracted functions reproduce known-good coordinate outputs *captured from the current implementation first*, and require a `npm run dev` visual smoke test of: canonical expand/collapse, "+N more" paging (zoom must not jump), and a two-paper comparison overlay before merging. Keep the comparison-side extraction (Step 6) entirely separate so a layout regression can't be conflated with a comparison-state regression.
