# Diagnosis: "+ more nodes" resets zoom/viewport

## TL;DR

There are **two** viewport-reset paths. One was already fixed; the other is the live culprit.

1. **Renderer pagination path** (`pageCluster`/`pageCompCluster`) — **already fixed** in commit `5b631e0`, with a passing regression test. It no longer calls `cy.fit()`. ✅
2. **React `ResizeObserver → resetView()` path** in `GraphPanel.tsx:51-54` — **this is the remaining root cause.** It calls `cy.fit()` on *any* container size change, which throws away the user's zoom/pan. The headless renderer tests cannot catch this because jsdom has no real layout/`ResizeObserver`.

## Root-cause mechanism

**The reset chain:**
```
"+ more" tap
  → graphRenderer.wireInteractions tap handler (graphRenderer.ts:3216-3224)
  → pageCluster()  [graphRenderer.ts:1233]  ── FIT-FREE, correct ✅
  → nodes .show()/.position()  (more fanned-out content drawn)
        │
        └─(in a real browser, content/layout shift can tick the ResizeObserver)
              → GraphPanel ResizeObserver callback (GraphPanel.tsx:51-53)
              → rendererRef.current.resetView()  [graphRenderer.ts:3014-3019]
              → this.cy.resize(); this.cy.fit(visibleNodes, 60)   ← ZOOM/PAN DESTROYED
```

**Why the renderer fix alone wasn't enough:** `resetView()` is an unconditional `cy.fit()` with no zoom/pan guard. It's wired to a `ResizeObserver` that fires not only on genuine container resizes (dragging the LeftPanel/DetailPanel splitters — `Workbench.css` grid uses `gridTemplateColumns: ${leftWidth}px 1fr` / rows `${bottomHeight}px`, both stateful/draggable) but also on the layout perturbations and the **initial observe() callback** that cytoscape's own canvas insertion + content growth can produce. Every one of those calls `fit()` and snaps the user back to "fit everything."

This is exactly the landmine called out in `CODE_GRAPH.md:194`: *"After any graph mutation (expand, overlay), cytoscape may trigger its layout… Preserve zoom/pan via `cy.zoom()` + `cy.pan()` capture/restore."* The renderer methods now honor this; `resetView()` does **not**.

## Minimal fix

Make `resetView()` (the resize handler) **resize without re-fitting**, so it only adapts cytoscape to the new canvas dimensions and preserves the user's zoom/pan. Refit only on a genuine, intentional reset (initial load / new graph), which already happens in `renderGraph()`.

**`src/renderer/graphRenderer.ts` — `resetView()` (lines 3014-3019):**
```ts
resetView(): void {
  if (!this.cy) return;
  this.cy.resize();  // re-query container dimensions; preserves zoom/pan
  // Do NOT cy.fit() here — this runs on every ResizeObserver tick and would
  // discard the user's zoom/pan on "+ more" expansion and panel drags (Issue #9).
}
```
Add a separate explicit `fitView()` for the cases that genuinely should refit (keep the old fit body), and call it only where a refit is wanted.

**`src/components/GraphPanel.tsx` — ResizeObserver (lines 51-54):** keep calling `resetView()` (now fit-free). Optionally debounce, but not required for correctness.

This is the smallest change: one method loses its `fit()`, the resize observer keeps adapting the canvas, and zoom/pan survive every "+ more" tap and every splitter drag.

**Alternative (more conservative)** if you want resize-driven refit *only* when the user hasn't manually zoomed: track a `userHasAdjustedViewport` flag (set in `zoomBy` and on cytoscape `zoom`/`pan` user events), and have `resetView()` skip `fit()` when it's set. More code, same outcome; I'd recommend the simpler version above unless product wants auto-fit on window resize specifically.
