# Refactor Plan: Splitting `src/renderer/graphRenderer.ts` (3302 lines)

## What's actually in the file

**Lines 30–66** — Types & constants: `RenderOptions`, `DEFAULT_RENDER_OPTIONS`, `NodeType`, `GROUP_META`, `ALL_GROUPS`, `MAX_FAN_SIZE`
**Lines 73–145** — Pure helper functions: `deriveNodeType`, `buildChildrenMap`, `buildEdgeTypeMap`, `formatEdgeType`, `computeImportance`
**Lines 148–624** — `buildStylesheet()` — one ~475-line pure function returning the Cytoscape stylesheet
**Lines 630–3302** — `CytoscapeGraphRenderer` — the god-class: 87 methods + 36 instance fields

The class is the real god-object, but it's also the part with shared mutable state (36 `private` fields all interdependent), so it's the *hardest* to split. The free functions above it are low-risk, high-value extractions.

## (2) The seam to split along

**Extract the pure, stateless code out of the class's orbit — do NOT carve up the stateful class yet.** Two target modules, plus a barrel:

```
src/renderer/
  graphRenderer.ts          ← keeps CytoscapeGraphRenderer class (stays the import path for GraphPanel)
  graph/
    graphTheme.ts           ← GROUP_META, ALL_GROUPS, MAX_FAN_SIZE, buildStylesheet()   (~480 lines, zero state)
    graphHelpers.ts         ← deriveNodeType, buildChildrenMap, buildEdgeTypeMap,
                              formatEdgeType, computeImportance, NodeType,
                              RenderOptions, DEFAULT_RENDER_OPTIONS                       (~110 lines, pure)
```

Why this seam and not "split the class into ClusterRenderer / OverlayRenderer / DragController":
- The 87 methods share **36 mutable fields** (`expandedNodes`, `childrenMap`, `cy`, `clusterPage`, …). Cluster methods, overlay methods, and drag methods all read and write the same `this.cy` / `expandedNodes` / `groupedNodes`. Splitting them into separate classes means threading that state through constructors or a shared context object — high risk, large diff, easy to introduce subtle state-desync bugs in zoom/pan or expand/collapse.
- The free functions (lines 30–624) have **zero `this` references**. `buildStylesheet` alone is ~480 lines of pure config — extracting it removes ~15% of the file with near-zero risk.

This split solves the stated problem (god-object, high fan-in) at the **module** level: `deriveNodeType` is imported by `noveltyClassifier`, so moving it breaks the existing circular import (see risk below). The class fan-in is unchanged because only `GraphPanel` uses it, and via a 6-method surface that needs no restructuring.

## (4) Top risk

**The pre-existing circular import between `graphRenderer.ts` ⇄ `noveltyClassifier.ts`.** Today `graphRenderer.ts:24` imports `classifyAllNodes` from `noveltyClassifier`, and `noveltyClassifier.ts:14` imports `deriveNodeType`/`NodeType` back. Vite/ESM tolerates this now only because of evaluation-order luck. When you move `deriveNodeType` into a new module, you must move it **cleanly to a leaf module** (`graphHelpers.ts` that imports nothing from `graphRenderer` or `noveltyClassifier`) and repoint *both* consumers there (steps 3 + 5). If you instead leave a re-export in `graphRenderer.ts` permanently, you preserve the cycle and risk a `deriveNodeType is undefined`-at-init failure that won't show in unit tests (which import the symbol directly) but **will** surface in the bundled app where module init order differs — i.e., a green `npm test` with a broken `npm run dev`. Mitigation: `graphHelpers.ts` must import only from `../types/*`; verify with `npm run build` (not just `npm test`) after steps 3 and 4, and do the live `npm run dev` smoke test before claiming done.

**Secondary risk:** `buildStylesheet`'s output is consumed by `renderGraph` → `cytoscape({ style })`; a copy-paste slip (the many `as unknown as number` casts on lines 616–620) would silently change node sizing rather than throw. The render-path test in `graphRenderer.test.ts` plus the UI smoke test are the guard.
