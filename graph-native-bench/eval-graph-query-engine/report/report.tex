\section*{Executive summary}
We evaluated a \native{graph-native} coding-agent harness against (a) a \textbf{plain} agent
(Read/Grep/Bash) and (b) the same agent with a code graph attached over \mcp{MCP} (graph-as-a-tool),
across three benchmarks on the same real codebase: a deterministic \textbf{retrieval oracle} (8 tasks),
a live agent \textbf{A/B} on the Claude subscription, and a \textbf{hardcore LLM-judged} suite of 6
diverse reasoning tasks (refactor planning, bug diagnosis, dead-code safety, data-flow tracing,
security audit, API migration). The graph engine is one local binary indexing the repo in
\textbf{1.1\,s} (1{,}267 nodes / 5{,}622 edges, TS+TSX+Py+JS).

\textbf{Headline.} Across the hardcore suite, \native{graph-native wins on mean quality (8.97/10),
is the cheapest arm, and has the best quality-per-dollar (8.2)} --- the ``better \emph{and} cheaper''
result. But \textbf{no single harness dominates}: the winner is predictable from \emph{task structure}.
Graph-as-a-tool (\mcp{MCP}) wins open-ended, search-shaped work (diagnosis, audit) where interactive
querying beats a static preamble; \native{graph-native} wins structured-traversal work (data-flow,
migration) where the answer is a known graph walk; plain stays competitive only on thorough-sweep tasks.

\section{The three harnesses}
The independent variable is \emph{wiring only} --- same model (\texttt{claude -p}, Claude
subscription, no API key), same tasks, same budget.
\begin{itemize}
\item \textbf{plain} --- agent with Read/Grep/Glob/Bash. Retrieval happens \emph{in} the agent.
\item \mcp{graph-MCP} --- same agent + the codegraph MCP server (\texttt{codegraph\_*} tools).
Retrieval happens in the agent, \emph{in-context}, over the raw graph.
\item \native{graph-native} --- the harness pre-runs the \emph{ranked, test-demoted} blast radius and
injects it before the agent's first token. Retrieval + ranking happen in deterministic harness code.
\end{itemize}
The win, when it occurs, is the \textbf{relocation of ranking out of the model's context into the
harness} --- not a better tool or model.

\subsection{Per-task findings (the answer to ``which harness for which task'')}
\begin{itemize}
\item \textbf{H1 refactor-plan} --- graph arms tie (9.0) $\ge$ plain (8.5); \native{native cheapest}
(\$0.71, 7 turns vs 13). Both graph arms named the dependents more completely.
\item \textbf{H2 bug-diagnosis} --- \mcp{graph-MCP wins (8.3)} $>$ native (7.1) $\gg$ plain (5.4).
Diagnosis needs a cross-file trace; interactive graph queries suit hypothesis-chasing better than a
static preamble.
\item \textbf{H3 dead-code-safety} --- plain $=$ \native{native (9.5)} $>$ MCP (7.5). All arms reached
the \emph{same correct} keep/delete verdicts (the React/Flask dispatch traps); native won on
calibration depth (a spec-aware ``confirm the feature is abandoned before deleting'' caveat).
\item \textbf{H4 data-flow} --- \native{native wins (9.8)}. Pre-injected structure ordered the
multi-hop frontend$\to$backend trace.
\item \textbf{H5 security-audit} --- \mcp{graph-MCP wins (10.0)}; all three found the
\texttt{pickle.load} RCE sink at \texttt{loader.py:112}. A shallow 2-hop sink is findable by any
careful agent --- graph not decisive.
\item \textbf{H6 api-migration} --- \native{native wins (8.6)} and is cheapest; the pre-injected caller
set is the edit list.
\end{itemize}

\section{Inside the three harnesses --- the actual agent process (case study: H2)}
To make the mechanism concrete, we re-ran H2 (diagnose why expanding the graph resets the zoom) with

from the captured transcripts (\texttt{Skill} = invoke the systematic-debugging skill; \texttt{CG} =

\subsection*{\textnormal{\normalsize \textbf{plain} --- 18 turns, 16 tool calls (6 Grep, 7 Read, 2 Bash)}}
\begin{trace}
\stp{1.}{Skill}{invoke \texttt{systematic-debugging}.}
\stp{2.}{Read CG}{opens \texttt{CODE\_GRAPH.md} first to localize --- it points at \texttt{graphRenderer.ts:64-66} and \texttt{GraphPanel.tsx}.}
\stp{3.}{Read}{\texttt{graphRenderer.ts} --- spots \texttt{pageCluster} (line 1227) with an Issue-\#9 comment.}
\stp{4.}{Read}{\texttt{GraphPanel.tsx} --- finds the \texttt{ResizeObserver $\to$ resetView()} at line 51.}
\stp{5--9.}{Grep $\times$3, Read $\times$1}{hunts \texttt{resetView|wireInteractions|pageCluster|.fit(} to trace how ``+ more'' is routed; reads \texttt{wireInteractions}.}
\stp{10--12.}{Bash, Read, Bash}{\texttt{grep -n repositionComparisonGraph}; reads it; \texttt{git log} --- learns commit \texttt{5b631e0} already fixed the pagination path.}
\stp{13--16.}{Grep $\times$3, Read $\times$1}{checks tests + \texttt{expandNode}; final grep \texttt{cy.zoom()|savedZoom|preserveViewport} confirms \emph{no} viewport-capture helper exists.}
\stp{$\Rightarrow$}{}{\textbf{quality 5.4} --- correct family of answer, but the most churn and least precise about the exact reset path.}
\end{trace}

\subsection*{\textnormal{\normalsize \mcp{\textbf{graph-MCP} --- 18 turns, 16 tool calls (6 Grep, 8 Read, 1 Bash)}}}
\begin{trace}
\stp{1--2.}{Skill, Read CG}{invoke skill; read \texttt{CODE\_GRAPH.md} (the repo mandates it) --- localizes Issue \#9.}
\stp{3--4.}{Grep $\times$2}{\texttt{more nodes|expand}, then \texttt{cy.(zoom|pan|fit|layout)} to find every camera call.}
\stp{5--6.}{Read $\times$2}{\texttt{graphRenderer.ts} + \texttt{GraphPanel.tsx} --- flags the \texttt{ResizeObserver} + the \texttt{isMore} click handler.}
\stp{7--10.}{Grep, Read $\times$3}{traces the exact ``+ more'' tap path through \texttt{wireInteractions} $\to$ \texttt{pageCluster}.}
\stp{11--13.}{Grep $\times$2, Bash}{checks \texttt{repositionComparisonGraph} + node-click dispatch; \texttt{git log} finds the prior \#9 fix.}
\stp{14--16.}{Read $\times$2, Grep}{reads \texttt{App.tsx}; final grep for zoom/pan-preservation confirms none exists --- the conclusive evidence.}
\stp{$\Rightarrow$}{}{\textbf{quality 8.3} (best) --- enumerated \emph{every} \texttt{fit()}-calling path and produced a complete 10-item re-test list.}
\end{trace}

\subsection*{\textnormal{\normalsize \native{\textbf{graph-native} --- 14 turns, 12 tool calls (\textbf{2} Grep, 8 Read, 1 Bash)}}}
\begin{trace}
\stp{1.}{Skill}{invoke \texttt{systematic-debugging}.}
\stp{2.}{Read}{\textbf{skips \texttt{CODE\_GRAPH.md}} --- the injected GRAPH CONTEXT already named the surface, so it goes \emph{straight} to \texttt{GraphPanel.tsx}.}
\stp{3.}{Grep}{one targeted \texttt{more nodes|expand|loadMore} to find the pagination entry.}
\stp{4--9.}{Read $\times$6}{reads \texttt{graphRenderer.ts} regions in sequence --- confirms \texttt{pageCluster} is already \texttt{fit}-free, then finds \texttt{toggleCluster}/\texttt{toggleNodeExpansion} \emph{do} call \texttt{cy.fit()} (the live bug), and verifies \texttt{repositionComparisonGraph} is pan-neutral.}
\stp{10.}{Grep}{second (and last) grep \texttt{cy.fit|resetView|viewport} to confirm the architecture.}
\stp{11--12.}{Read, Bash}{reads the test file; \texttt{ls + git log} to confirm prior fixes.}
\stp{$\Rightarrow$}{}{\textbf{quality 7.1}, \native{fewest turns (14) and searches (2 greps)} --- it spent its budget \emph{verifying} the injected map, not discovering it.}
\end{trace}

\noindent\textbf{What the real process shows.} All three ran the same skill and reached a defensible
diagnosis, but the \emph{paths diverge exactly at retrieval}: \textsc{plain} and \mcp{graph-MCP} both
\emph{open by reading \texttt{CODE\_GRAPH.md}} and then run a 6-grep hunt to discover the change
surface; \native{graph-native} \emph{skips that entirely} --- the harness already injected the ranked
surface, so the agent dives straight into the source and uses only \native{2 greps} to verify. That
is the efficiency mechanism, observed literally: \emph{4 fewer searches and 4 fewer turns because the
discovery work was already done before the model's first token}.

\noindent\textbf{Honest caveat (disclosed).} In this captured run the \mcp{graph-MCP} agent did
\textbf{not} invoke its \texttt{codegraph\_*} MCP tools --- it solved H2 with text search like
\textsc{plain}, so its measured profile matched plain's (tool use is non-deterministic across runs;
its 8.3 win came from thoroughness, not from firing the graph tools here). The contrast that
reproduces deterministically is \native{graph-native}'s skipped discovery loop, which follows by
construction from injecting the surface.

\section{Retrieval oracle \& live A/B (the reproducible scale layer)}
A deterministic oracle scored what each harness \emph{delivers} to the agent (file-list F1 @ top-15)
over 8 tasks spanning impact, caller-enumeration, data-flow, dead-code, refactor-triage, and
test-selection. A live \texttt{claude -p} A/B then confirmed the oracle predicts real-agent behavior.

\begin{figure}[h]\centering

\noindent\textbf{Live validation.} On the firehose impact task (\texttt{PaperGraph}: 59 files, 52 of

cheaper} (\$0.608 vs \$0.673) --- the oracle's prediction held end-to-end. (Disclosed: the oracle's
absolute F1 is an upper bound; real agents under-enumerate, so trust the oracle for arm \emph{ordering}.)

\section{Method integrity \& disclosed negatives}
\begin{itemize}
\item \textbf{Blind, rubric-based judging.} The judge is a fresh \texttt{claude -p} with no tools and
no repo access; it grades fenced answer text only, so it cannot favor a harness it cannot see.
\item \textbf{A judge bug we caught and fixed.} The first judge build scored a \emph{correct}
9.8-quality security answer as 0.0 --- long code-bearing answers triggered a prompt-injection-style
refusal. Fixed via system-prompt role-pinning, fenced-data framing, and robust JSON extraction;
re-verified at 9.8.
\item \textbf{An honest self-correction.} Our first read of H3 (``MCP over-trusted the graph'') was
wrong: all arms got the verdicts right; the gap was calibration tone. Corrected in the record.
\item \textbf{Operational reality.} The initial 30-call run hit the monthly API spend limit (HTTP 429)
after task 4; we hardened the harness to survive 429 and resume, then completed H5/H6 without re-paying.
\item \textbf{Threats to validity.} $n$ is small (6 hardcore tasks, 8 oracle tasks, 1 repo, 1 run/arm);
results are directional. LLM-judge bias (length/confidence/self-preference) is mitigated, not
eliminated. graph-MCP beating native on H2/H5 and plain tying on H3/H5 are kept in --- a benchmark
where the graph always wins would be rigged.
\end{itemize}

\section{Conclusion}
A code graph helps coding agents most when the harness \textbf{queries and ranks it before the agent's
first token} --- the relocation of ranking into deterministic code. On this real codebase that
\native{graph-native} design is the best average and the most cost-efficient harness. The deeper,
honest result is that the \emph{right} graph integration is task-shaped: pre-injection for structured
traversals, interactive graph tools for open-ended diagnosis, and plain search where a thorough sweep
already suffices. There is no universal best harness --- but there is a predictable one.
